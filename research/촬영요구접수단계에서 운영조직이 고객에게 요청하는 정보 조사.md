# 촬영요구접수단계에서 운영조직이 고객에게 요청하는 정보 조사

기준일: 2026-03-07

## 결론 요약

한국 위성영상 운영조직이 촬영요구 접수 단계에서 고객에게 요구하는 정보는 대체로 4개 묶음으로 정리된다.

1. `누가 요청하는가`
2. `어디를 언제 어떤 조건으로 찍을 것인가`
3. `어떤 형태의 산출물로 받을 것인가`
4. `왜 요청하는가, 어떤 우선순위로 처리해야 하는가`

공개적으로 입력 항목이 가장 명확한 한국 사례는 다음 두 가지다.

- `상용 저궤도 지구관측위성`: SI Imaging Services `SpaceEye` 신규촬영(New Tasking Order)
- `정지궤도 기상위성 특별관측`: 국가기상위성센터 `GK2A AMI Rapid Scan(ARS)` 요청

반면 `국토위성 1호`나 `군 정찰위성`은 공개 문서에서 일반 고객용 촬영요청 입력 화면까지 확인되지는 않았다. 따라서 아래 보고서는 `공개적으로 확인 가능한 입력 항목`만 정리했다.

## 조사 범위와 해석 원칙

- 포함: 운영기관이 고객 또는 이용기관에게 실제로 입력하도록 요구하는 항목
- 포함: 회원 등록 단계에서 요구하는 사용자 식별 정보
- 제외: 내부 운용자가 뒤에서 계산하는 궤도/자세/메모리/다운링크 계획 정보
- 제외: 비공개 군 운용 절차

또한 아래 구분을 명확히 했다.

- `직접 확인`: 주문 안내 페이지, 사용자 매뉴얼, 등록 양식에 명시된 항목
- `합리적 추론`: 요청 현황 표나 절차 설명으로부터 역으로 확인되는 항목

## 한눈에 보는 표

| 운영 사례 | 고객이 제공해야 하는 정보 묶음 | 공개 확인 수준 | 비고 |
| --- | --- | --- | --- |
| `SI Imaging Services SpaceEye` 신규촬영 | AOI, 촬영기간, 센서/위성/촬영모드, 구름량, 오프나디어/입사각, 제품 레벨, 제품 타입, 우선순위, 전달매체, 활용분야, 추가설명, 최종사용자 정보, 라이선스 동의 | 직접 확인 | 한국 상용 EO/SAR 주문 사례 중 가장 구체적 |
| `국가기상위성센터 GK2A ARS` 특별관측 | 등록기관 정보, 연락처, 등록 이메일, 요청 제목, 신청 목적, 고정/추적 관측 유형, 시작/종료 시각, 경도/위도, 관측 지속시간 | 직접 확인 + 일부 추론 | 일반 대국민 서비스가 아니라 등록된 기관용 특별관측 |
| `국토위성센터/국토지리정보원` | 공개 문서상 촬영계획 전달 주체는 확인되나, 일반 외부고객 입력 항목은 미확인 | 미확인 | 내부 또는 기관 간 절차는 보이지만 고객 입력화면은 공개 확인 어려움 |

## 1. 상용 저궤도 위성 사례: SI Imaging Services SpaceEye

### 1-1. 고객 식별과 계정 등록 단계

SpaceEye 사용자 매뉴얼에 따르면 사용자는 먼저 `user registration form`을 내려받아 `orderdesk@si-imaging.com`으로 보내 계정을 등록해야 한다. 공개 문의 양식에서는 다음 연락처 항목이 확인된다.

- 이름
- 회사명
- 국가
- 전화번호
- 이메일
- 문의 유형
- 문의 내용

즉, 상용 주문의 가장 바깥 단계에서는 최소한 `요청 주체의 신원`과 `연락 가능한 채널`을 먼저 확보한다.

### 1-2. 신규 촬영 주문에서 직접 요구하는 정보

SpaceEye 사용자 매뉴얼과 주문 안내 페이지를 합치면, 신규 촬영 단계에서 고객이 입력하거나 선택하는 항목은 아래처럼 정리된다.

#### A. 촬영 대상과 기간

- `AOI(Area of Interest)`
- `촬영 유효기간 / image acquisition window`
- AOI 표현 방식
  - 원: 중심 위도/경도, 반경
  - 사각형: 4개 꼭짓점 좌표
  - 파일 업로드: SHP, KML, KMZ

즉, 운영조직은 먼저 `어디를 찍을지`와 `언제까지 찍어야 하는지`를 요구한다.

#### B. 센서와 획득 조건

- `Sensor type`
- `Satellite`
- `Imaging mode`
- `Cloud coverage`
- `Off-nadir angle`(광학)
- `Incidence angle`(SAR)
- `Orbit direction`
- `Monitoring count`
- `InSAR / repeat acquisition 여부`

즉, 같은 위치라도 `광학인지 SAR인지`, `일반 촬영인지 반복 관측인지`, `촬영 각도 제약이 있는지`까지 고객이 명시해야 한다.

#### C. 산출물과 처리 옵션

- `Product level`
- `Product type`
- `Product data type`(SAR)
- `Ancillary`
- `File format`
- `Delivery media`

운영조직은 촬영 자체만이 아니라 `어느 수준까지 보정된 제품을 어떤 매체로 받을지`도 함께 요구한다.

#### D. 업무 목적과 우선순위

- `Priority`
- `Application fields`
- `Additional description`

이 정보는 단순 참고가 아니라, 긴급성 판단, feasibility 조정, 상업 우선순위 배정에 직접 쓰인다.

#### E. 사용자 및 법적 정보

- `End user information`
- `KOMPSAT EULA 동의`

매뉴얼에는 주문 전에 최종사용자 정보를 입력하고, 라이선스에 동의해야 한다고 나온다. 다만 공개된 매뉴얼 텍스트만으로는 end user information 내부 세부 항목 이름까지 전부 보이지는 않는다.

### 1-3. 실무적으로 보면 운영조직이 고객에게 묻는 질문

SpaceEye 사례를 실무 문장으로 바꾸면 운영조직은 고객에게 사실상 아래를 묻는다.

- 어느 지역을 찍고 싶은가
- 언제까지 필요한가
- 광학과 SAR 중 무엇을 원하는가
- 구름량, 오프나디어, 입사각 제한은 얼마인가
- 한 번만 찍을지 반복 촬영할지
- 어떤 처리 레벨과 파일 포맷이 필요한가
- 긴급 주문인지 일반 주문인지
- 어떤 업무에 쓸 것인지
- 최종 사용자는 누구인지

즉, `위치 + 시간 + 촬영 제약 + 제품 요구 + 사용자 식별`이 핵심이다.

## 2. 정지궤도 기상위성 특별관측 사례: 국가기상위성센터 GK2A ARS

### 2-1. 이 서비스의 성격

GK2A AMI Rapid Scan(ARS)은 일반 공개 촬영주문이 아니라, 등록된 기관 사용자가 특정 지역과 시간에 대해 `고빈도 특별관측`을 요청하는 서비스다. 공식 설명에는 사용자가 dedicated web page를 통해 요청한다고 되어 있고, 사전 등록이 필요하다고 적혀 있다.

### 2-2. 사전 등록 단계에서 요구하는 정보

공개된 `GK2A AMI Rapid Scan Service Registration Form` 검색 결과 기준으로, 등록 양식에는 다음이 포함된다.

- `Nation`
- `Organization`
- `Contact point`
  - Name
  - Affiliation
  - E-mail
  - Phone number
- `Account name`
- `Name of person responsible`
- `Signature`

즉, 이 서비스는 개인 단위 즉시 주문이 아니라 `기관 자격 확인 + 담당자 인증`이 먼저다.

### 2-3. 실제 특별관측 요청 단계에서 요구하는 정보

NMSC 특별관측 설명 페이지와 Rapid Scan request manual 검색 결과를 합치면, 요청 시 사용자가 제공하는 항목은 다음과 같이 정리된다.

- `registered email address`
- `desired start time`
- `desired end time`
- `Subject`
- `Purpose of Application`
- `fixed observation` 또는 `tracking observation` 선택
- `Longitude & Latitude`
- `Observation Duration`

여기서 중요한 점은 관측 유형에 따라 입력값 구조가 달라진다는 것이다.

- `fixed observation`: 특정 위경도 중심의 고정 지역을 일정 시간 집중 관측
- `tracking observation`: 이동하는 현상 추적 중심 관측

검색 결과에 노출된 특별관측 현황 표에는 아래 컬럼도 확인된다.

- 국가
- 제목
- 영상종류
- 관측 시작일
- 관측 종료일
- 위도
- 경도
- 처리 상태

이 중 `처리 상태`는 운영시스템이 부여하는 값이고, 나머지 상당수는 요청 메타데이터로 보는 해석이 합리적이다.

### 2-4. 실무적으로 보면 운영조직이 고객에게 묻는 질문

GK2A ARS 사례를 실무 문장으로 바꾸면 운영조직은 고객 기관에 아래를 요구하는 셈이다.

- 어느 기관이 요청하는가
- 요청 책임자와 연락 가능한 이메일/전화는 무엇인가
- 특별관측의 제목과 목적은 무엇인가
- 고정 관측인지 추적 관측인지
- 어느 위치를 중심으로 볼 것인가
- 언제 시작하고 언제 끝낼 것인가
- 총 얼마 동안 고빈도 관측이 필요한가

즉, 상용 EO 주문보다 단순하지만 `기관 식별 + 특별관측 목적 + 시간/위치`가 필수다.

## 3. 공개 자료 기준 공통 입력 항목 체계

위 두 사례를 합치면, 한국 운영조직이 촬영요구 접수 단계에서 고객에게 요구하는 정보는 다음 표준 구조로 요약할 수 있다.

| 구분 | 고객 입력 항목 | 왜 필요한가 |
| --- | --- | --- |
| 요청 주체 식별 | 기관명, 담당자명, 소속, 이메일, 전화, 국가, 계정명 | 권한 확인, 결과 통보, 계약/책임 주체 확인 |
| 촬영 대상 | AOI, 위도/경도, 대상 지역 파일(SHP/KML/KMZ) | 센서 시야와 궤도 통과 가능성 계산 |
| 촬영 시점 | 시작일, 종료일, acquisition window, observation duration | 궤도/시간 슬롯 배정 |
| 센서 선택 | 광학/SAR, 위성명, 촬영모드, 고정/추적관측 유형 | 임무계획과 탑재체 설정 결정 |
| 품질 제약 | 구름량, 오프나디어, 입사각, 반복관측 횟수 | feasibility와 가격/우선순위 계산 |
| 제품 요구 | 제품 레벨, 제품 타입, 데이터 타입, 파일 포맷, 전달매체 | 후처리 및 배포 방식 결정 |
| 업무 맥락 | 활용 분야, 신청 목적, 추가 설명, 긴급도/우선순위 | 상충 시 우선 배정, 내부 승인, 정책 판단 |
| 법적/계약 | 최종사용자 정보, 라이선스 동의 | 이용 제한, 재배포 제한, 책임 소재 확정 |

## 4. 공개 자료상 확인되지 않은 부분

### 4-1. 국토위성 1호

국토위성센터 공개 자료에서는 산출물 종류와 활용 체계는 확인되지만, 외부 고객이 입력하는 `촬영요청 신청 화면`이나 `신청서 항목`은 이번 조사 범위에서 공식적으로 확인하지 못했다.

즉, `서비스 결과물과 활용 분야`는 공개돼 있지만, `고객 입력 필드`는 공개 자료상 미확인이다.

### 4-2. 군 정찰위성

고객 입력 항목, 요청 절차, 승인권자, 우선순위 규칙은 공개되지 않는다. 따라서 공개 보고서에서는 다루지 않는 것이 맞다.

## 최종 정리

촬영요구 접수 단계에서 운영조직이 고객에게 묻는 정보의 본질은 아래 한 문장으로 정리할 수 있다.

`누가`, `어디를`, `언제`, `어떤 제약조건으로`, `어떤 제품 형태로`, `어떤 목적으로` 원하는지를 받아야만 이후의 feasibility 계산과 임무계획 작성이 가능하다.

한국 공개 사례를 기준으로 보면:

- `상용 지구관측위성 주문`은 가장 상세하며, 촬영 조건과 산출물 요구까지 촘촘히 받는다.
- `기상위성 특별관측 요청`은 기관 자격과 특별관측 목적, 위치, 시간 중심으로 받는다.
- `국토위성`은 운영 주체 간 역할은 공개돼 있지만, 고객 입력 항목은 공개적으로 확인되지 않았다.
