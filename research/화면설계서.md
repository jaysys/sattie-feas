# K-Sattie Feasibility Analysis Console 화면설계서

## 1. 문서 개요
- 문서명: 화면설계서(역설계)
- 시스템명: K-Sattie Feasibility Analysis Console
- 작성목적: 현재 개발 완료 화면을 기준으로 고객 제출용 UI 설계 산출물을 확정한다.
- 작성기준: 실제 구현 소스(`src/static/index.html`, `src/static/app.js`, `src/static/styles.css`) 반영
- 작성일: 2026-03-08

## 2. 적용 범위
- 웹 단일 화면 콘솔(요청 목록, 요청 상세, 타당성분석리포트, 수행계획 후보 시뮬레이션, 결과 카드/리스트)
- 모달 2종
1. 새 요청 생성 모달
2. 외부 요청번호 매핑 모달

## 3. 화면 구성 개요
메인 화면은 `좌측 요청목록 + 우측 상세작업영역` 2단 구조로 구성한다.

1. 상단 Hero 영역
- 타이틀: `K-Sattie Feasibility Analysis Console`
- 설명문: 위성촬영요청 등록/후보 시뮬레이션/판정 관리 안내 문구
- 지표 카드: `기준 API`, `실시간 상태`, `요청 수`

2. 좌측 사이드바
- 섹션명: `요청 목록`
- 우측 상단 `+` 아이콘 버튼으로 `새 요청 생성` 모달 호출
- 요청 엔티티 리스트 표시/선택

3. 우측 본문 영역
- 요청 헤더 카드(요청명, 설명, 요청코드 태그, 센서 태그)
- 요청 헤더 우측 하단 `✎` 아이콘 버튼으로 `외부 요청번호 매핑` 모달 호출
- `고객요청정보상세 및 정책정보` 접기/펼치기 패널
- 그룹 패널 `타당성분석리포트`
- 그룹 패널 `요청명 기반 수행계획 후보별 시뮬레이션`
- 하단 2열 보조결과 패널(모델분해결과/완화 제안, 탈락 사유/자원 및 다운링크)

## 4. 상세 화면 명세

### 4.1 요청 헤더 영역
- 표시항목
1. 요청 제목
2. 요청 설명
3. 요청코드 태그
4. 센서유형 태그
- 기능
1. `✎` 버튼 클릭 시 외부 요청번호 매핑 모달 오픈

### 4.2 고객요청정보상세 및 정책정보 패널
- 기본 접힘(`is-collapsed`) 상태
- 토글 버튼 `+`/접힘 상태 전환
- 내부 구성
1. 고객 요청 정보(내부요청코드, 외부요청번호, 기관, 요청자, 기간, AOI, 센서/모드)
2. 고객 요청 제한(구름/오프나디르/입사각/태양고도/커버리지/마감/모니터링횟수)
3. 정책 적용 정보(정책명, 우선순위, 집계상한, 반복품질/간격/입사각 일관성, SLA)

### 4.3 타당성분석리포트 그룹
- 그룹 타이틀: `타당성분석리포트`
- 하위 2개 서브패널
1. Proposal 요약
2. 후보 검증 테이블

#### 4.3.1 Proposal 요약 패널
- eyebrow: 요청코드 동적 표기
- 제목 규칙: `'요청명'에 대한 Feasibility Analysis Summary`
- 상단 요약카드 3종
1. 최종 판정
2. 전체 확률
3. 최적안
- 하단 개요 테이블(우측 값은 문장형 동적 텍스트)
1. 누적 성공확률
2. 예상 첫 촬영기회
3. 예상 촬영기회 소진 수
4. 집계 반영 후보 수
5. 요구 반복 횟수
6. 반복 요구 충족
7. 품질 반영 시도 수
8. 입사각 일관성 충족
9. 일관성 반영 시도 수
10. 반복 간격 충족
11. 간격 충족 시도 수
- 문장형 출력 규칙
1. `예/아니오` 판정은 `"예"로 판정되었습니다.` 형식 적용
2. 수치형은 `xx건이 집계에 반영되었습니다.` 등 단위 포함 문장 출력
- 스크롤 정책
1. 테이블 전체가 보이면 내부 스크롤 미표시(`summary-table-shell overflow visible`)

#### 4.3.2 후보 검증 테이블 패널
- eyebrow: 요청코드 동적 표기(기존 `후보 테이블` 대체)
- 제목 규칙: `'요청명'에 대한 수행계획 후보별 검증 결과`
- 컬럼
1. 후보
2. 상태
3. 촬영기회
4. 오프나디르
5. 입사각
6. 데이터량
7. P(total)

### 4.4 수행계획 후보별 시뮬레이션 그룹
- 제목 규칙: `'요청명'에 대한 수행계획 후보별 시뮬레이션`
- 구성
1. 후보 목록 관리
2. 후보 입력값 등록/수정
3. 후보 평가 결과

#### 4.4.1 후보 목록 관리
- 섹션명: `수행계획 후보 목록`
- 버튼
1. 새 후보 준비
2. 선택 후보 삭제

#### 4.4.2 후보 입력값 등록 및 수정
- 주요 입력군
1. 후보 메타(코드, 제목, 설명, 상태, 순서, 기준안 여부)
2. 촬영조건(센서, 우선순위, 면적, 윈도우, 촬영기회 시각)
3. 광학조건(구름, 오프나디르, 태양고도)
4. SAR조건(입사각)
5. 커버리지/데이터량/저장/다운링크/전력/열
- 버튼
1. 선택 후보 다시 불러오기
2. 후보 저장
3. 저장 후 검증 실행
- 동작
1. 입력 변경 시 현재평가 자동 스케줄링(지연 후 시뮬레이션)
2. 센서 타입 변경 시 광학/SAR 필드 표시 전환

#### 4.4.3 후보 평가 결과
- eyebrow: 선택 후보코드 동적 표기
- 제목 규칙: `'요청명'의 '수행계획후보명' 수행계획 평가 결과`
- 상단 카드
1. 현재 판정
2. 현재 확률
- 상세
1. 현재 사유
2. 현재 점검 결과
3. 실행 이력(전체/추천안 실행만 필터)

### 4.5 하단 분석 보조 패널
- 좌상: 제목 규칙 `'요청명' 모델분해결과`
- 우상: 제목 규칙 `'요청명' 완화 제안`
- 좌하: 제목 규칙 `'요청명' 후보별 탈락 사유`
- 우하: 제목 규칙 `'요청명' 자원 및 다운링크`
- 보조설명 툴팁(`?`) 제공: 모델분해결과/완화 제안

## 5. 팝업(모달) 설계

### 5.1 새 요청 생성 모달
- 호출 위치: 좌측 `요청 목록` 상단 `+`
- 타이틀: `새 요청 생성`
- 입력항목: 요청제목/설명/정책/우선순위/기관/요청자/기간/마감/모니터링/AOI/외부연계코드
- 저장버튼: `새 요청 저장`
- 닫기 방식: `×`, 오버레이 클릭, `Esc`
- 여백 정책: 상하좌우 충분한 패딩(`modal-overlay`, `modal-panel`) 적용

### 5.2 외부 요청번호 매핑 모달
- 호출 위치: 요청 헤더 우측 하단 `✎`
- 타이틀: `외부 요청번호 매핑`
- 목록 + 편집폼 제공
- 제약사항(현행)
1. `외부 요청번호` 필드 읽기전용
2. 저장버튼 비활성
3. 제출 시 `외부 요청번호는 읽기 전용입니다.` 메시지 표시
- 닫기 방식: `×`, 오버레이 클릭, `Esc`

## 6. 동적 문구/표기 규칙
- 요청명은 주요 제목에서 작은따옴표 포함 표기
1. `'서울 AOI 광학 촬영 요청건' ...`
- 후보명 포함 제목도 작은따옴표 적용
1. `'요청명'의 '후보명' 수행계획 평가 결과`
- `에 대한` 문구 사용 범위
1. Summary/검증결과 제목에는 유지
2. 후보 시뮬레이션 제목은 `... 후보별 시뮬레이션` 형식 사용
- boolean 판정형 문구는 `"예"` 또는 `"아니오"`를 따옴표 포함 텍스트로 표기

## 7. 입력/상태 시각 규칙
- 읽기전용/비활성 입력 필드 배경: 연회색
- 커서: `not-allowed`
- 모달 오픈 시 본문 스크롤 잠금(`body.modal-open`)
- 실시간 상태 값
1. `/health` 정상 시 `정상`
2. 실패 시 `오프라인`

## 8. 반응형/레이아웃 규칙
- 데스크톱: 2열 중심
- 중간폭(1120px 이하): 주요 그리드 1열 전환
- 모바일(680px 이하): 패널/모달 패딩 축소, 라운드 축소, 폼 1열

## 9. 화면별 API 연계(구현 기준)
1. `GET /health` 실시간 상태
2. `GET /requests` 요청 목록
3. `POST /requests` 새 요청 생성
4. `GET /requests/{request_code}` 요청 상세/리포트 로드
5. `PATCH /requests/{request_code}` 요청 수정
6. `POST /requests/{request_code}/cancel` 요청 취소
7. `GET /requests/{request_code}/result-access` 결과 접근 가이드
8. 후보 관리
- `GET/POST/PATCH/DELETE /requests/{request_code}/request-candidates...`
- `POST /requests/{request_code}/request-candidates/{candidate_code}/simulate`
- `POST /requests/{request_code}/simulate-candidate-input`
9. 외부참조
- `GET/POST/PATCH/DELETE /requests/{request_code}/external-refs...`
- 단, 화면에서는 읽기전용 정책으로 저장 동작 제한

## 10. 사용자 시나리오(대표)
1. 요청 생성
- 요청 목록 `+` 클릭 → 새 요청 생성 모달 입력 → 저장 → 요청 자동 로드
2. 후보 시뮬레이션
- 후보 선택/입력 변경 → 현재평가 확인 → 저장 후 검증 실행 → 실행이력 확인
3. 리포트 확인
- 타당성분석리포트에서 최종판정/확률/최적안/후보검증결과 확인

## 11. 특기사항
- 본 문서는 구현 역설계 기준이며, 정책적으로 비활성화된 기능(외부요청번호 수정)은 UI 상 명시적으로 잠금 처리되어 있다.
- API는 수정/취소/결과접근 가이드를 제공하나, 화면의 노출 범위는 운영 정책에 따라 제한될 수 있다.
