## 1) 핵심 약어/전문용어 (A-Z)

| 용어/약어 | 영문 | 설명 |
|---|---|---|
| ADCS | Attitude Determination and Control System | 위성의 자세를 결정·제어하는 시스템. 목표 AOI를 촬영할 수 있는지(기동 가능성)와 직결된다. |
| AOI | Area of Interest | 촬영 대상 영역. 폴리곤/WKT/면적/중심점/방향각과 함께 요청의 핵심 입력값이다. |
| API | Application Programming Interface | 시스템 간 기능 호출 인터페이스. 본 시스템은 REST API(FastAPI)로 제공된다. |
| Assured/Priority/Standard | 서비스 티어 | 상용 위성영상 서비스에서 주문 우선권·마감·SLA를 구분하는 계약 등급. |
| Candidate | 수행계획 후보 | 요청 하위의 가상 검토안. 각 후보는 별도 입력값과 평가/실행 이력을 가진다. |
| Cloud Cover | 구름량 | 광학 촬영 가능성에 직접 영향. `max_cloud_pct` 제약으로 관리된다. |
| Conflict/Deconfliction | 충돌/충돌해소 | 기존 task/downlink 예약과의 시간·자원 충돌을 검토하고 해소하는 과정. |
| Coverage Ratio | 커버리지 비율 | AOI 중 실제 촬영으로 덮는 비율. `predicted/required`로 비교한다. |
| CRUD | Create/Read/Update/Delete | 데이터 기본 처리 방식. 본 시스템의 요청/후보/외부참조 API 설계 기본. |
| DIS | Data Ingestion System (관행) | 외부/수신 데이터를 수집·검증·적재하는 계층을 의미하는 경우가 많다. 기관별 정의 차이 가능. |
| Downlink | 하향링크 | 위성→지상국 데이터 전송. contact window, 링크율, 예약충돌에 의해 가능성이 달라진다. |
| EO | Earth Observation | 지구관측. 광학/열적외/레이더 등 관측 임무 전반을 포괄한다. |
| ERD | Entity Relationship Diagram | 데이터 엔터티와 관계를 표현한 모델. 본 프로젝트는 요청-후보-실행이력 중심 구조. |
| Feasibility | 타당성(촬영가능성) | 요청 조건, 궤도/환경/자원/정책/충돌을 종합해 수행 가능성을 판정하는 과정. |
| FK | Foreign Key | 테이블 간 참조 무결성 키. 요청↔후보, run↔결과 등 관계를 보장한다. |
| GeoTIFF | GeoTIFF | 지리좌표/투영 메타데이터를 포함하는 TIFF 기반 래스터 포맷. |
| GIS | Geographic Information System | 공간정보 저장/분석/시각화 체계. AOI/궤도/접근분석과 결합된다. |
| HDF5 | Hierarchical Data Format 5 | 대용량 과학데이터 계층형 저장 포맷. SAR/기상/과학제품에 자주 사용. |
| HH/HV/VH/VV | SAR Polarization | SAR 편파 채널 조합. 요청 편파와 센서모드 지원 편파 일치 여부를 검증한다. |
| ICD | Interface Control Document | 시스템 인터페이스 정의서. 외부연계 필드/코드 체계 합의에 사용된다. |
| Incidence Angle | 입사각 | SAR 관측에서 핵심 기하 파라미터. 품질/해석 가능성과 반복촬영 일관성에 영향. |
| Ingest | 적재 | 원천 데이터/메타데이터를 저장소에 수집·정규화하는 단계. |
| IRPS | Image Receiving & Processing System | 영상 수신 및 처리 시스템(관행 용어). 수신-복원-처리-산출물화 흐름을 담당. |
| JSON | JavaScript Object Notation | API 요청/응답의 기본 데이터 표현 형식. |
| KML/KMZ/SHP | 공간파일 포맷 | AOI 교환에 사용하는 대표 형식들. |
| KOMPSAT | Korea Multi-Purpose SATellite | 다목적실용위성 계열 명칭. 본 연구 문서에 사례로 등장. |
| LEO | Low Earth Orbit | 저궤도. 지구관측 위성이 주로 운용되는 궤도 영역. |
| Local Time Window | 현지시각 선호창 | 요청자가 선호하는 촬영 현지시각 범위. 벗어나면 품질/판독 리스크로 반영. |
| MAPS | Mission Analysis and Planning System | 임무분석/계획 시스템(문헌 용어). 이벤트 예측, 스케줄링, 명령계획 기능 포함. |
| MCS | Mission Control System | 임무통제 시스템(문헌/상용 용어). |
| MOC | Mission Operation Center | 위성 운영 통제 조직/시스템. |
| Monitoring Count | 모니터링 횟수 | 반복 촬영 요구 횟수. 반복충족/간격/일관성 검증의 기준값. |
| NetCDF | Network Common Data Form | 과학·기상 데이터 교환 포맷. |
| OGC | Open Geospatial Consortium | 공간정보 표준화 기구. SPS/EO Tasking Extension 등 관련 표준 제공. |
| Off-nadir | 오프나디르 | 위성 정하점에서 벗어난 관측각. 클수록 기동/왜곡/품질 제약이 증가할 수 있다. |
| Opportunity | 촬영기회 | 실제 촬영 가능한 시간 구간 후보(접근/기하/운영 조건 기반). |
| Orthorectified/Ortho | 정사보정 | 지형/센서 기하를 보정해 지도좌표 기준으로 정렬한 영상 제품. |
| P(total) | 후보 총확률 | 후보 단건의 종합 확률(기하·환경·자원·다운링크·충돌 조정 반영). |
| Payload | 탑재체 데이터 | 관측센서가 생성한 영상/관측 데이터. TT&C 링크와 구분되는 고속 downlink 경로를 쓴다. |
| PK | Primary Key | 테이블 기본키. |
| PMS | Product Management System (관행) | 처리된 영상 산출물의 메타데이터/버전/배포를 관리하는 계층으로 쓰이는 경우가 많다. 기관별 정의 차이 가능. |
| Polarization | 편파 | SAR 전파 송수신 편파 조합(HH/HV/VH/VV). |
| Priority Tier | 우선순위 등급 | `STANDARD/PRIORITY/URGENT` 등. 정책 검증과 충돌 해소 우선순위 판단에 사용. |
| Proposal | 타당성 제안서 요약 | 요청 전체 누적확률, 첫 촬영기회, 최적안, 완화옵션 등을 묶은 결과 요약. |
| Recorder Backlog | 저장 백로그 | 위성 저장장치에 누적된 미전송 데이터량. |
| Repeat Acquisition | 반복 촬영 | 단건이 아닌 다회 촬영 요구. 반복 횟수/간격/입사각 일관성 검증 필요. |
| SAR | Synthetic Aperture Radar | 능동형 레이더 영상. 구름 영향은 작지만 입사각/지형 layover/shadow 제약이 중요하다. |
| SLA | Service Level Agreement | 서비스 수준 협약. 마감·우선순위·처리수준 등 운영 정책으로 반영된다. |
| Snapshot | 스냅샷 | 특정 시점 외부 데이터 버전(궤도/기상/자원/지상국). 재현성 확보에 중요. |
| SSO | Sun-Synchronous Orbit | 태양동기궤도. 동일 지역을 유사한 태양시각에 관측하기 유리한 궤도. |
| Stripmap/Spotlight | SAR/광학 모드 예시 | 관측 모드에 따라 해상도/폭/데이터량/운용제약이 달라진다. |
| Tasking | 촬영 임무화 | 사용자 요청을 위성 촬영 명령 가능한 계획으로 변환하는 과정. |
| Terrain Layover/Shadow | 지형 중첩/음영 | SAR에서 지형으로 인한 왜곡/가림 현상. 판독 품질 리스크로 반영된다. |
| TLE | Two-Line Element set | 궤도 예측에 사용하는 표준 궤도요소 형식(문헌 용어). |
| TT&C | Telemetry, Tracking and Command | 위성 상태 송신/추적/명령 링크. 대용량 payload downlink와 분리 운용되는 것이 일반적. |
| Uplink | 상향링크 | 지상국→위성 명령 전송. |
| UTC | Coordinated Universal Time | 표준 시각. 본 시스템 datetime 저장/검증 기준. |
| WKT | Well-Known Text | 공간 기하(POINT/POLYGON 등) 텍스트 표현 형식. |
| X-band/S-band | 주파수 대역 | 일반적으로 X-band는 고속 payload downlink, S-band는 TT&C에 쓰이는 경우가 많다. |

## 2) 시스템/아키텍처 용어

| 용어 | 설명 |
|---|---|
| 요청(Request) | 고객 또는 외부시스템이 제출한 촬영 요구 단위. 내부 요청코드(`request_code`)로 관리. |
| 외부 요청번호 매핑 | 내부 요청과 외부시스템 요청번호를 연결하는 구조(`request_external_ref`). |
| 기준안(Baseline Candidate) | 후보 비교 기준축. 요청당 1건만 유지되며 반복 일관성 계산에 우선 사용. |
| 현재 평가(Current Evaluation) | 후보 입력값을 저장하지 않고 즉시 계산한 평가 결과. |
| 저장 실행(Save & Run) | 후보 입력 저장 후 실행 이력(`request_candidate_run*`)으로 남기는 검증 실행. |
| 요청 전체 집계(Request-level Aggregation) | 후보 집합을 정책상 상한 내에서 반영해 누적 성공확률·첫 촬영기회 등을 계산. |
| 완화 제안(Relaxation Recommendation) | 확률/판정 개선을 위해 조정 가능한 파라미터 변경 제안. |
| 실행 트리거(Trigger) | 수동 실행/추천안 반영/분리후보 생성 등 실행 사유 메타정보. |
| 동적 계산(Dynamic Evaluation) | seed 고정 결과 조회가 아니라 현재 후보입력 기반으로 재계산하는 방식. |

## 5) 판정 사유(Reason Code) 사전

| 사유 코드 | 단계 | 설명 |
|---|---|---|
| `MIN_ORDER_AREA_NOT_MET` | POLICY | 후보 면적이 정책 최소 주문면적 미달 |
| `ORDER_CUTOFF_MISSED` | POLICY | 주문 마감시각 초과 |
| `PRIORITY_UPGRADE_REQUIRED` | POLICY | 후보 우선순위가 요청/계약 우선순위보다 높음(상향 필요) |
| `SERVICE_POLICY_PRIORITY_MISMATCH` | POLICY | 요청 우선순위가 정책 허용범위 초과 |

## 9) 혼동이 잦은 용어 정리

| 용어 | 주의점 |
|---|---|
| 요청 시간창 vs 촬영기회 | 요청 시간창은 고객 요구 기간, 촬영기회는 실제 촬영 가능 시간 구간 후보다. |
| 후보 현재 평가 vs 저장 실행 결과 | 현재 평가는 비저장 즉시 계산, 저장 실행 결과는 이력 테이블에 남는 검증 결과다. |
| 후보 확률 vs 요청 전체 확률 | 후보 확률은 단건(`p_total_candidate`), 요청 전체 확률은 누적(`overall_probability`)이다. |
| TT&C vs Payload Downlink | TT&C는 제어/상태 링크, payload downlink는 대용량 영상 전송 링크다. |
| DIS/PMS | 기관마다 약어 정의가 달라질 수 있어 프로젝트 ICD/용어집에서 공식 정의를 우선한다. |
